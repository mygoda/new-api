# 账单页面 v2 PRD

**版本**: v0.3 / 2026-05-09
**作者**: 平台研发
**状态**: 待评审
**焦点**: **只讲账单页面怎么展示、怎么看**——不涉及权限/角色/团队/订阅/充值

---

## 1. 一句话目标

让用户进 `/console/billing` 这一个页面就能回答 4 个问题:

1. 我这个月**总共**花了多少钱?
2. 钱**主要**花在哪几个模型?
3. 哪几次请求**特别贵**(异常)?
4. 我能**翻到**任意一笔的详细流水吗?

---

## 2. 页面信息架构(自顶向下)

```
┌─────────────────────────────────────────────────────────────┐
│  /console/billing                                           │
│                                                             │
│  [时间筛选: 本月 ▾]  [模型筛选 ▾]  [Token 筛选 ▾]  [⬇ 导出] │
│                                                             │
│  ┌───────────┬───────────┬───────────┬───────────┐         │
│  │ 本月消费  │ 预测月底  │ 余额      │ 调用次数  │ ← 卡片区
│  │ ¥1,234   │ ¥3,500    │ ¥5,000    │ 1,847     │         │
│  │ ↑ +12%   │ vs ¥3,100 │           │ ↑ +8%     │         │
│  └───────────┴───────────┴───────────┴───────────┘         │
│                                                             │
│  [按模型] [按厂商] [按时间] [按 Token] [异常请求] [流水明细]│ ← Tab 区
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │            (当前 Tab 内容,见下文 §4 ~ §9)         │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**核心交互原则**:

- **任何一处数字都能点击钻取**(饼图扇区点 → 钻进该模型的明细;时间线某一天点 → 钻进那天的流水)
- **Tab 切换不丢筛选**(顶部时间/模型/Token 在 Tab 间共享)
- **明细流水是终点**:任何钻取最终都进同一个明细表

## 3. 顶部:总览卡片区

4 张卡片,横向排列,移动端 2×2 布局。

| 卡片 | 主数字 | 次数据 | 取值口径 |
|------|--------|--------|----------|
| **本月消费** | ¥1,234 | ↑ +12%(同比上月同期) | Σ 当前月已结算消费,USD→CNY 按系统汇率 |
| **预测月底** | ¥3,500 | vs ¥3,100(上月) | 当前 daily 速率 × 当月剩余天数 |
| **余额** | ¥5,000 | (用户查自己钱包) | `users.quota / QuotaPerUnit` |
| **调用次数** | 1,847 | ↑ +8%(同比) | Σ 请求条数 |

**视觉细节**:

- 主数字字号 28-32px,粗体
- 同比变化文字小一号:正向 ↑ 红(更花钱) / 负向 ↓ 绿(更省) / 平的灰色
- 「预测月底」如果是月底/月初(数据少)显示「样本不足」灰色
- 「余额」点击 → 跳转 `/console/topup`

## 4. Tab 1:按模型(默认)

> 用户最常问的:**钱花在哪几个模型**。

```
┌────────────────────────────┬─────────────────────────────────┐
│                            │ Top 10 模型                     │
│       (饼图,8 扇区)       │ ┌──────────────────────────────┐│
│                            │ │ 1. gpt-4o          ¥520 42% ││
│       gpt-4o 42%           │ │ 2. claude-sonnet   ¥320 26% ││
│       claude  26%          │ │ 3. gemini-pro      ¥150 12% ││
│       gemini  12%          │ │ 4. doubao-seed     ¥110  9% ││
│       其他    20%          │ │ 5. deepseek-v3     ¥ 60  5% ││
│                            │ │ ...                          ││
│                            │ └──────────────────────────────┘│
└────────────────────────────┴─────────────────────────────────┘
```

**饼图**:

- 最多 8 扇区,第 8 扇是「其他」聚合(避免太碎)
- 鼠标 hover 一个扇区:tooltip 显示「模型名 / 金额 / 调用次数 / 平均单价」
- 点击扇区:**整个 Tab 切到「该模型的子视图」**——
  - 顶部 breadcrumb:`按模型 / gpt-4o`
  - 该模型的:时间趋势线图 + 调用分布(prompt/completion token 直方图)+ 该模型的明细流水
  - 顶部「× 返回」回到饼图

**Top 10 表格**:

- 列:**模型名 / 调用次数 / 总 token / 总金额 / 占比 / 平均单价(¥/1K token)**
- ❌ **不展示**:channel / 渠道厂商 / 上游 endpoint
- 整行可点击(等同点饼图扇区)
- 排序:默认按金额降序,可切换按调用次数

## 5. Tab 2:按厂商

> 跟「按模型」类似,但是按**模型品牌方**聚合(OpenAI / Anthropic / Google / 字节 / 阿里 / DeepSeek / Moonshot / 智谱 / 其他)。

**为什么单独一个 Tab**:用户脑里其实分两层——
- "我用了 GPT 系列还是 Claude 系列?"(品牌层)→ 这个 Tab
- "我具体用的是 gpt-4o 还是 gpt-4o-mini?"(模型层)→ 上一个 Tab

视觉同饼图 + 表格,扇区数固定 9 个(8 厂商 + 其他)。

**厂商分组规则**:前端写死一份正则,前后端共用,不依赖后台 channel 配置。

```
OpenAI       → /^(gpt-|o1-|chatgpt-|dall-e-|whisper-|tts-|text-embedding-)/i
Anthropic    → /^claude-/i
Google       → /^(gemini-|palm-|bison-|embeddings-gecko)/i
ByteDance    → /^(doubao-|seedance|seedream)/i
Alibaba      → /^(qwen-|wanx-|dashscope-|embedding-v\d)/i
DeepSeek     → /^deepseek-/i
Moonshot     → /^(moonshot-|kimi)/i
ZhipuAI      → /^(glm-|chatglm-)/i
Other        → 兜底
```

> 用户问的:**消费走势,哪天在烧钱**。

```
┌────────────────────────────────────────────────────────────┐
│  [日 / 周 / 月]  [最近 30 天 ▾]  ☐ 显示去年同期对比       │
│                                                            │
│   400 ┤                                                    │
│   300 ┤        ╱╲                                          │
│   200 ┤    ╱╲ ╱  ╲    ╱╲                                   │
│   100 ┤  ╱╲╱  V    ╲╱  ╲╱╲                                 │
│     0 └─────────────────────────                           │
│       1   5   10   15   20   25   30                       │
│                                                            │
│   ▏▎▎▏▎▍▏▍▍▎▎▎▍▎▍▏▍▎▍▎▎▍▎▏▍▎▎▎▍ ← 调用量柱状图(次轴)    │
└────────────────────────────────────────────────────────────┘
```

**线图**:

- 主轴(左):每日消费金额(¥)
- 次轴(右):每日调用次数
- 双线叠加,颜色区分(消费实线、调用次数虚线)
- 鼠标 hover 一个数据点:tooltip 显示「日期 / 消费 / 调用次数 / Top 模型」
- 点击数据点:**钻进该日的明细流水**
- 「显示去年同期」勾上后叠加灰色对比线

**辅助线**(可选):

- 30 天移动平均(灰色虚线)
- 每日预算上限(如果用户设了月预算 / 30,显示水平红线)

**粒度切换**:

- 日(默认):时间轴 = 最近 30 天
- 周:时间轴 = 最近 12 周
- 月:时间轴 = 最近 12 个月

## 7. Tab 4:按 Token / 业务标签

> 用户问的:**哪个项目最烧钱**。

按用户的 API Token Name 聚合(token 是用户在 `/console/token` 创建的命名 key,通常对应一个项目/环境)。

```
┌──────────────────────────────────────────────────────────┐
│  Token Name                调用    消费    占比   走势   │
│  ──────────────────────────────────────────────────────  │
│  prod-frontend             820    ¥520    42%    📈+15% │
│  staging-test              420    ¥220    18%    📉 -5% │
│  research-notebook         180    ¥180    14%    📈+30% │
│  default                   1200   ¥150    12%    →  ±0% │
│  ...                                                     │
└──────────────────────────────────────────────────────────┘
```

- 列:**Token Name / 调用次数 / 消费 / 占比 / 走势(上月对比)**
- 走势用箭头 + 百分比,最后 7 天 vs 前 7 天
- 整行点击 → 钻进「该 Token 的所有请求」
- 没有命名 Token 的请求归到「default」

> **业务标签**(`tags` JSON 数组,用户在请求 metadata 里传)是更细维度,可以放在 Token 下面再聚合,**P1 加**,P0 先做 Token 维度。

## 8. Tab 5:异常请求

> 用户问的:**哪几次请求特别贵 / 出问题了**。

按规则识别 4 类异常,排序展示:

| 异常类型 | 规则 | 标签颜色 |
|----------|------|----------|
| 🔥 **高消耗** | 单次消费 > 该用户本月 P99 单次 × 2 | 红 |
| ♻️ **频繁重试** | 同一个 prompt(基于 hash)5 分钟内 > 3 次 | 橙 |
| 📜 **超长上下文** | prompt_tokens > 该模型 80% 上下文窗口 | 黄 |
| ❌ **失败请求** | 状态非 2xx 但仍计费(部分上游会扣费) | 灰 |

```
┌──────────────────────────────────────────────────────────┐
│  时间        模型         类型        Token   消费       │
│  ──────────────────────────────────────────────────────  │
│  10:24:31   gpt-4o      🔥高消耗     45K     ¥18.30    >│
│  10:18:02   claude-sonnet ♻️重试 ×4   2K       ¥8.80     >│
│  09:55:11   gemini-pro  📜超长ctx     180K    ¥12.50    >│
│  ...                                                     │
└──────────────────────────────────────────────────────────┘
```

- 每行末尾 `>` 展开:看具体的请求 ID、prompt 前 200 字符(脱敏)、推荐操作
- 推荐操作示例:
  - 高消耗:「相同任务可考虑 deepseek-v3,预计省 60%」
  - 频繁重试:「检查 prompt 格式,可能上游 token 上限被触发」
  - 超长上下文:「考虑用 RAG 切分」

> P0 不需要"推荐操作"的智能算法,**只展示规则识别 + 用户原因猜测的提示文案**;真正的推荐 P3 再做。

## 9. Tab 6:流水明细

> 用户问的:**给我看一笔笔的请求**。

```
┌────────────────────────────────────────────────────────────────┐
│  时间          模型         厂商      Token      In/Out  消费 状态│
│  ─────────────────────────────────────────────────────────────  │
│  14:32:11    gpt-4o        OpenAI    prod-fe    1.2K/450 ¥0.18 ✓│
│  14:31:55    gpt-4o        OpenAI    prod-fe    980/120  ¥0.12 ✓│
│  14:30:02    claude-sonnet Anthropic prod-fe    2.1K/800 ¥0.34 ✓│
│  14:29:18    gemini-pro    Google    research   8.5K/2K  ¥0.95 ⚠│
│  ...                                                           │
│                                                                │
│  ◀ 1  2  3  ... 47 ▶   [50 条/页 ▾]   显示 1-50 / 2,341         │
└────────────────────────────────────────────────────────────────┘
```

**列定义**(用户视角):

| 列 | 来源 | 示例 |
|----|------|------|
| 时间 | `created_at`(Doris) | 14:32:11(同日)/ 03-15 14:32(跨日) |
| 模型 | `model_name` | gpt-4o |
| **厂商** | 由前端正则按模型名映射(同 §5) | OpenAI |
| Token | `token_name` | prod-fe / default |
| 输入/输出 | `prompt_tokens` / `completion_tokens` | 1.2K / 450 |
| 消费 | `quota / QuotaPerUnit` | ¥0.18 |
| 状态 | `is_success`(2xx)/ 错误 | ✓ / ⚠ / ✗ |

**列定义,用户视角下永不展示**(本期最重要的展示约束):

- ❌ `channel_id` / `channel_name`(后台路由信息,平台机密)
- ❌ 上游真实 endpoint URL
- ❌ 原始上游错误日志(若状态为 ⚠/✗,只显示模板化文案如「上游服务暂时不可用」)
- ❌ 任何渠道维度的 vendor 字段(model 的 vendor 与 channel 的 vendor 是两个概念)

**单行展开**(点击行):

```
┌─ Request ID: req-abc123def456 ────────────────────┐
│  请求时间    2026-05-09 14:32:11.234              │
│  耗时        1.45s                                │
│  模型        gpt-4o                               │
│  上下文      1,234 tokens                         │
│  输出        450 tokens                           │
│  缓存读      0 tokens                             │
│  缓存写      0 tokens                             │
│  分组倍率    × 1.0                                │
│  消费        ¥0.18                                │
│                                                   │
│  [复制 Request ID]  [查看 Prompt(脱敏前 500 字)] │
└───────────────────────────────────────────────────┘
```

**翻页 / 筛选**:

- 默认 50 条/页,可选 20/100/200
- 顶部全局筛选(时间/模型/Token)对此 Tab 同样生效
- 此 Tab 额外加一个内联快搜:输入 Request ID 直接定位

## 10. 全局筛选器

顶部一行,所有 Tab 共享:

| 筛选 | 选项 | 默认 |
|------|------|------|
| **时间范围** | 今天 / 本周 / 本月(默认)/ 最近 7 天 / 最近 30 天 / 自定义 | 本月 |
| **模型** | 多选,从用户实际用过的模型动态生成 | 不筛 |
| **Token** | 多选,从用户的 Token Name 列表生成 | 不筛 |
| **厂商**(可选) | 9 个品牌组多选 | 不筛 |

筛选状态写入 URL query,刷新 / 分享链接保留状态。

## 11. 导出

右上角「⬇ 导出」按钮:

- **CSV**:当前筛选 + 当前 Tab 的全量数据(不分页),用户拿去 Excel 或入财务
- **PDF 月度账单**:整月汇总(总览 + 按模型 + 按厂商 + Top 100 流水),抬头自填(不开发票,但能给老板看)
- 移动端不显示 PDF 选项(只 CSV)

字段同 §9 表格,**仍不含 channel**。

## 12. 空态 / 加载态 / 错误态

| 场景 | 表现 |
|------|------|
| 用户当月零消费 | 总览卡显示 ¥0,Tab 区显示插画+「本月还没有调用,去 [模型广场](/marketplace) 看看?」 |
| 数据加载中 | 总览卡 skeleton + Tab 区 spinner + 灰色掩码,**保留上次成功的数据**避免闪烁 |
| Doris 查询超时(>3s) | Tab 区显示「数据查询繁忙,稍后重试」+ 重试按钮,但**总览卡降级用 MySQL 兜底**(余额/调用次数能从 logs 表算) |
| 时间范围超过 90 天 | 提示「明细只保留最近 90 天,更早的数据请下载月度账单」 |

## 14. 数据来源(简述,与展示有关)

| 展示用的数据 | 来源 | 备注 |
|--------------|------|------|
| 顶部总览卡 | Doris `request_logs` 聚合 | 余额从 MySQL `users` |
| 按模型 / 按厂商 / 按时间 / 按 Token | Doris `request_logs` GROUP BY | 厂商分组前端做 |
| 异常请求 | Doris `request_logs` WHERE 规则 | 规则在后端,可调 |
| 流水明细 | Doris `request_logs` ORDER BY ts DESC | 用户视角字段过滤 |
| 单行展开 | Doris `request_logs` 详情 | 同上,channel 字段在 API 层 strip |

> **⚠️ 关键约束**:不要让前端拿到含 `channel_*` 的字段后再决定不展示。**API 层就要 strip 干净**,前端不应有此知识。

## 15. 不在本期范围(本 PRD 不涉及)

- 团队 / 子账户 / 配额(独立模块)
- 充值 / 订阅 / 自动续费(归 `/console/topup` `/console/subscription`)
- Admin 账单后台(`/admin/billing`,字段不同的另一套页面,独立 PRD)
- 发票(明确不做)
- 上游成本毛利分析(明确不做)
- 推荐降级模型(规则识别 P0,智能推荐 P3)
